# AI PM 的技术边界

> 最常被问的问题：AI PM 需要会写代码吗？答案是：不必会，但必须能读懂。

## 核心定位

**AI PM 的技术职责是决策质量，而非实现能力。**

你需要：
- 能判断技术方案的合理性
- 能与工程师高效沟通（不被忽悠）
- 能估算技术成本和风险
- 能提出合理的技术约束条件

你不需要：
- 自己写 LLM 应用
- 精通 PyTorch / 调参
- 深入理解矩阵运算

## 必须懂的技术知识

### 1. 模型能力边界

知道模型"能做什么、不能做什么"是产品决策的前提。

| 模型能力 | 可用于 | 不适合 |
|---------|--------|--------|
| 文本生成 | 内容创作、代码补全、报告生成 | 需要精确数字计算 |
| 推理 | 分析、决策建议、诊断 | 要求 100% 准确率 |
| 检索增强（RAG） | 知识库问答、客服 | 需要实时数据（分秒级） |
| 多模态 | 图片理解、UI 分析 | 视频实时流处理（成本高） |

### 2. 核心性能指标

作为 PM，你需要能看懂并解读以下指标：

| 指标 | 含义 | 产品影响 |
|------|------|---------|
| Latency（延迟） | 首字符响应时间 | 用户体验，>3s 显著流失 |
| TTFT | Time to First Token，首 Token 时间 | 流式输出的感知速度 |
| Throughput | 每秒生成 Token 数 | 高并发下的响应能力 |
| Context Length | 单次处理的最大 Token 数 | 决定能处理多长的文档 |
| Cost/Token | 每 Token 的费用 | 单次调用和月度成本 |

### 3. 常见技术风险

| 风险类型 | 具体表现 | PM 应对 |
|---------|---------|---------|
| 幻觉（Hallucination） | 模型编造事实 | 增加来源引用、人工审核高风险输出 |
| 提示词注入 | 用户通过输入劫持模型行为 | 输入过滤、权限隔离 |
| 成本失控 | 复杂 Agent 任务 Token 暴涨 | 设置 Token 预算上限、任务超时终止 |
| 数据泄露 | 用户数据进入模型训练 | 确认模型方不用 API 数据训练 |

### 4. 部署模式选择

| 模式 | 特点 | 适用场景 |
|------|------|---------|
| API 调用（云端） | 低成本、快速上线 | 大多数产品 |
| 私有化部署 | 数据不出域，成本高 | 金融/医疗等合规要求高 |
| 边缘推理（本地） | 低延迟、隐私好，性能有限 | 手机 App、离线场景 |
| 混合部署 | 本地小模型 + 云端大模型 | 成本与性能平衡 |

## 不需要深入的技术细节

以下内容了解概念即可，**不需要能手写实现**：

- Transformer 数学细节（注意力矩阵运算）
- 训练过程（梯度下降、反向传播）
- 模型量化的具体技术（GPTQ、AWQ）
- 向量数据库的索引算法（HNSW、IVF）
- CUDA / GPU 调优

## 技术沟通能力：三个场景练习

### 场景 1：评审技术方案

工程师说："我们用 RAG + Reranker 实现知识库问答"

你应该问：
- "召回精度预计能到多少？有 Baseline 吗？"
- "Reranker 会增加多少延迟？用户能接受吗？"
- "知识库更新频率怎样？全量重新 Embedding 成本是多少？"

### 场景 2：向上汇报技术风险

不要说："模型会出幻觉，我们没办法。"

应该说："我们在高风险输出（医疗建议、财务数据）上增加了人工审核环节，并在 UI 上明确标注'AI 生成，请核实'，将幻觉风险控制在可接受范围内。"

### 场景 3：估算 AI 功能成本

老板问："加个 AI 摘要功能，成本怎样？"

你应该能算出：
```
假设：
- DAU 10万，30% 使用摘要功能 = 3万次/天
- 平均每次：3000 Token 输入 + 500 Token 输出
- 使用 GPT-4o-mini：$0.15/百万输入 + $0.6/百万输出

日成本 = 3万 × (3000×0.15 + 500×0.6) / 1,000,000
       = 30000 × (0.00045 + 0.0003)
       = 30000 × 0.00075 = $22.5/天 ≈ $675/月
```

## 技术学习路径建议

```
基础层（1个月）
├── 了解 Transformer 原理（不需要会写）
├── 掌握 Token / Context / Temperature 等核心概念
└── 会用 OpenAI / Claude API 写简单 Demo

进阶层（2-3个月）
├── 理解 RAG 架构，搭过一个简单知识库
├── 用过 Dify / Coze 等低代码平台做原型
└── 能看懂 LangChain / CrewAI 的基本代码逻辑

实战层（持续）
├── 参与真实 AI 产品的需求评审
├── 定期阅读模型厂商的技术博客
└── 跟踪 AI 产品竞品的技术动态
```

## 面试高频问题

**Q：你的技术背景怎样？怎么弥补技术短板？**

> "我的策略是'够用原则'：深入理解能力边界和风险，而不是实现细节。我会定期动手做原型（用低代码平台或 API），确保对技术有感性认知。遇到不懂的，我会直接问工程师，并把结论结构化后带回产品决策。"

**Q：你会写代码吗？**

> "我能写基础的 Python 和 JavaScript，可以用 Claude/GPT API 搭简单原型来验证想法。但我不会把精力花在成为工程师上——PM 的价值在于定义'做什么'和'为什么做'，而不是'怎么做'。"
