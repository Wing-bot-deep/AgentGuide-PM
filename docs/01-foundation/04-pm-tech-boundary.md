## 第一幕：技术方案评审会（8分钟）

### 场景还原

**时间**：项目启动会  
**参与者**：你（PM）、工程师小李、技术Leader老王

**小李的方案**：
```
技术选型：
- 模型：GPT-4o
- 架构：RAG（检索增强生成）
- 向量数据库：Pinecone
- 预计召回精度：70%
```

**老王**："方案看起来不错，PM你有什么问题吗？"

### 你的第一个问题："成本预估是多少？"

**为什么要问这个？**

因为AI功能的成本 = Token消耗 × 单价，不问清楚容易超预算。

**小李的回答**："呃...还没算..."

**你**："我们一起算一下。"

---

#### 知识点1：Token和成本估算

**Token是什么？**

Token是AI处理文本的最小单位：
- 1个Token ≈ 1个中文字
- 1个Token ≈ 0.75个英文单词

**为什么重要？**

Token是AI的计费单位，就像打车按公里收费一样。

**实战计算**

```
假设：
- DAU 5000，每人每天咨询3次 = 15000次/天
- 平均输入：200 Token（用户问题："我的订单什么时候到？"）
- 平均输出：300 Token（AI回答：300字的详细说明）
- GPT-4o：$2.5/百万输入Token，$10/百万输出Token

计算过程：
单次成本 = (200×2.5 + 300×10) / 1,000,000
         = (0.0005 + 0.003)
         = $0.0035

日成本 = 15000 × $0.0035 = $52.5
月成本 = $52.5 × 30 = $1575/月 ✅ 在预算内
```

**可复用工具：成本计算器（复制到Excel）**

```excel
=DAU × 使用频率 × (输入Token×输入单价 + 输出Token×输出单价) / 1000000 × 30
```

**会议继续**

**你**："月成本$1575，在预算内，但要留20%缓冲。"  
**老王**："不错，还有其他问题吗？"

---

### 你的第二个问题："召回精度70%意味着什么？"

**小李的回答**："就是能检索到相关文档的概率。"

**你追问**："那30%检索不到怎么办？用户体验会怎样？"

**小李**："呃...这个..."

---

#### 知识点2：RAG的召回精度

**什么是RAG？**

RAG = 检索增强生成，工作流程：
1. 用户提问："退货政策是什么？"
2. 系统从知识库检索相关文档
3. 把文档喂给模型，生成回答

**召回精度是什么？**

召回精度 = 成功检索到相关文档的概率

**行业标准**

| 召回精度 | 用户体验 | 真实数据 |
|---------|---------|---------|
| 85%+ | 优秀 | 用户满意度4.5/5 |
| 70-85% | 及格 | 用户满意度3.5/5，投诉率20% |
| <70% | 不及格 | 用户满意度<3.0，投诉率>30% |

**真实案例**

某公司RAG召回精度从70%优化到85%后：
- 用户满意度：3.2 → 4.3
- 投诉率：25% → 8%
- 转人工率：40% → 15%

**决策建议**

70%不够，要求工程师优化到85%+

**可复用工具：召回精度评估清单**

```
评审RAG方案时，逐项检查：
- [ ] 有测试集吗？（至少100条真实用户问题）
- [ ] 召回精度是多少？（目标85%+）
- [ ] 召回失败时怎么处理？（转人工/明确告知"未找到相关信息"）
- [ ] 有Baseline吗？（对比优化前后的效果）
```

**会议继续**

**你**："70%太低，能优化到85%吗？"  
**小李**："可以加Reranker，但会增加延迟和成本。"  
**你**："延迟增加多少？成本增加多少？"  
**小李**："延迟+0.5秒，成本+$50/月。"  
**你**："可以接受，总延迟控制在2秒内，总成本$1625/月。"

### 你的第三个问题："为什么选GPT-4o而不是GPT-4o-mini？"

**小李的回答**："GPT-4o能力更强。"

**你追问**："客服场景需要最强模型吗？mini便宜5倍，效果差多少？"

**小李**："这个...我没对比过..."

---

#### 知识点3：模型选型

**模型对比**

| 模型 | 输入价格 | 输出价格 | 速度 | 适用场景 |
|------|---------|---------|------|---------|
| GPT-4o | $2.5/百万 | $10/百万 | 快 | 复杂推理、代码生成 |
| GPT-4o-mini | $0.15/百万 | $0.6/百万 | 很快 | 简单任务、客服 |
| Claude Sonnet | $3/百万 | $15/百万 | 中 | 长文本理解 |
| Claude Haiku | $0.25/百万 | $1.25/百万 | 很快 | 简单任务、成本敏感 |

**决策框架**

```
客服场景特点：
- 80%是简单查询（退货政策、物流查询、账号问题）
- 不需要复杂推理
- 对速度要求高（<2秒）

→ 结论：用GPT-4o-mini够用
```

**成本对比**

```
用GPT-4o：
- 月成本：$1575 + Reranker $50 = $1625

用GPT-4o-mini：
- 输入：$0.15/百万（便宜16倍）
- 输出：$0.6/百万（便宜16倍）
- 月成本：$1625 / 16 ≈ $100 + Reranker $50 = $150

省下：$1475/月！
```

**可复用工具：模型选型决策树**

```
是否需要复杂推理（代码生成/长文档分析/多步推理）？
├─ 是 → GPT-4o / Claude Sonnet
└─ 否 → 是否对成本敏感？
    ├─ 是 → GPT-4o-mini / Claude Haiku
    └─ 否 → GPT-4o（平衡选择）
```

**会议继续**

**你**："建议先用GPT-4o-mini测试，如果效果不够再换GPT-4o。"  
**小李**："好的，我测试一下。"

**（2天后，小李测试结果）**

**小李**："测试了100条真实问题，GPT-4o-mini准确率92%，GPT-4o准确率95%，差距不大。"  
**你**："那就用mini，省$1475/月。"

---

### 会议结论

**最终方案**：
- 模型：GPT-4o-mini
- 架构：RAG + Reranker
- 召回精度目标：85%+
- 月成本：$150（远低于预算$2000）

## 第二幕：上线后的问题处理（8分钟）

### 场景还原

**时间**：上线2周后  
**问题**：客服反馈用户投诉"AI答非所问"，满意度从4.2降至3.5

**客服提供的3个典型Bad Case**：
1. 用户问"退货政策"，AI回答了"发货政策"
2. 用户问"会员等级"，AI说"未找到相关信息"（但知识库有）
3. 用户问"最新活动"，AI回答了3个月前的活动

---

### 你的诊断过程（完整展示）

#### 步骤1：分类问题

用"AI不准的5种类型"分类：

| Bad Case | 问题类型 | 原因分析 |
|---------|---------|---------|
| Case 1 | 理解偏差 | 意图识别错误，把"退货"理解成"发货" |
| Case 2 | 召回失败 | 知识库有答案但没检索到（Embedding质量差） |
| Case 3 | 知识过时 | 知识库没更新 |

**AI不准的5种类型**：
1. **幻觉**：编造事实（如引用不存在的法条）
2. **召回失败**：知识库有答案但没检索到
3. **理解偏差**：误解用户意图
4. **知识过时**：知识库没更新
5. **边界外问题**：超出AI能力范围

---

#### 步骤2：找工程师对齐

**你和小李的对话**：

**你**："小李，我们的Embedding模型是哪个？"  
**小李**："text-embedding-ada-002"  
**你**："这个模型对中文支持不太好，能换成text-embedding-3-large吗？"  
**小李**："可以，但成本会增加$20/月。"  
**你**："可以接受，召回率能提升多少？"  
**小李**："预计从85%提升到90%。"  
**你**："好，那Case 1的理解偏差怎么解决？"  
**小李**："可以增加意图识别模块，对模糊查询进行澄清式追问。"  
**你**："具体怎么做？"  
**小李**："用户问'退货政策'时，AI先确认：'您是想了解退货政策吗？'，用户确认后再回答。"  
**你**："开发需要多久？"  
**小李**："2天。"  
**你**："Case 3的知识过时怎么解决？"  
**小李**："建立每周自动更新机制，从官网抓取最新内容。"  
**你**："开发需要多久？"  
**小李**："3天。"

---

#### 步骤3：制定优化方案

| 问题 | 解决方案 | 成本 | 预期效果 | 优先级 |
|------|---------|------|---------|--------|
| 理解偏差 | 增加意图识别+澄清式追问 | 开发2天 | 理解偏差降低50% | P0 |
| 召回失败 | 换Embedding模型 | +$20/月 | 召回率85%→90% | P0 |
| 知识过时 | 建立每周自动更新机制 | 开发3天 | 知识时效性100% | P1 |

**可复用工具：Bad Case分析模板**

```markdown
【问题描述】
用户输入：退货政策是什么？
AI输出：我们的发货政策是...
期望输出：我们的退货政策是...

【问题分类】
☑ 理解偏差  ☐ 召回失败  ☐ 幻觉  ☐ 知识过时  ☐ 边界外

【根本原因】
意图识别错误，把"退货"理解成"发货"

【解决方案】
短期（本周）：增加澄清式追问
中期（2周）：优化意图识别模型
长期（1个月）：建立意图识别测试集

【预期效果】
理解偏差降低：30% → 15%
用户满意度提升：3.5 → 4.0
```

---

#### 步骤4：验证效果

**（1周后，优化上线）**

**数据对比**：

| 指标 | 优化前 | 优化后 | 提升 |
|------|--------|--------|------|
| 召回率 | 85% | 90% | +5% |
| 理解偏差率 | 30% | 15% | -50% |
| 用户满意度 | 3.5/5 | 4.2/5 | +0.7 |
| 投诉率 | 25% | 10% | -60% |

**你学到的**：
- 用"AI不准的5种类型"快速定位问题
- 用Bad Case分析模板结构化诊断

## 第三幕：成本失控预警（4分钟）

### 场景还原

**时间**：上线1个月后  
**问题**：运营反馈成本从$150飙升到$480

---

### 你的排查过程

#### 步骤1：查看监控数据

**成本监控看板**：

| 指标 | 预期 | 实际 | 状态 |
|------|------|------|------|
| DAU | 5000 | 5000 | ✅ 正常 |
| 日均调用次数 | 15000 | 15000 | ✅ 正常 |
| 平均输入Token | 200 | 200 | ✅ 正常 |
| 平均输出Token | 300 | **950** | ❌ 异常！ |
| 日成本 | $5 | $16 | ❌ 超预算3倍 |

---

#### 步骤2：找到根本原因

**你和小李的对话**：

**你**："输出Token从300涨到950，发生了什么？"  
**小李**："我看看日志...发现部分用户问'如何退货'，AI生成了2000字的详细教程。"  
**你**："为什么会这样？"  
**小李**："我们没限制输出长度，模型觉得越详细越好。"  
**你**："能限制输出长度吗？"  
**小李**："可以，限制最多500 Token。"  
**你**："会影响用户体验吗？"  
**小李**："不会，500 Token（约350字）足够回答大部分问题。"

---

#### 步骤3：优化方案

**方案**：限制输出最多500 Token

**成本对比**：

```
优化前：
- 平均输出：950 Token
- 月成本：$480

优化后：
- 平均输出：500 Token
- 月成本：$480 × (500/950) ≈ $252

省下：$228/月
```

**可复用工具：成本监控看板（Excel模板）**

```
【每日监控】
日期 | DAU | 调用次数 | 平均输入Token | 平均输出Token | 日成本
2024-01-01 | 5000 | 15000 | 200 | 300 | $5.0
2024-01-02 | 5000 | 15000 | 200 | 950 | $16.0 ⚠️

【预警阈值】
- 日成本 > 预算×1.2 → 🟡 黄色预警
- 日成本 > 预算×1.5 → 🔴 红色预警

【异常诊断】
如果日成本异常，检查：
1. DAU是否暴涨？
2. 调用频率是否增加？
3. 输入Token是否过长？
4. 输出Token是否过长？ ← 本次问题
```

---

### 你学到的

- 建立成本监控看板，每日检查
- 成本异常时，逐项排查：DAU、调用次数、输入Token、输出Token
- 限制输出长度是最简单的成本优化方式
