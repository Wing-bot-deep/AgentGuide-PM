# AI PM 技术能力自检与提升指南

> 最常被问的问题：AI PM 需要会写代码吗？答案是：不必会，但必须能读懂。本指南帮你明确技术边界，并提供可执行的提升路径。

## 一、快速自检：你处于哪个技术段位？

### 段位测试（3选1，选最接近你当前状态的）

**初级段位（技术小白）**
- [ ] 能说清楚 LLM、Agent、RAG 的基本概念
- [ ] 知道 Token 是什么，为什么它是成本单位
- [ ] 能区分 GPT-4 和 GPT-4o-mini 的使用场景

**高级段位（技术决策者）**
- [ ] 能独立评估技术方案的合理性，给出优化建议

- [ ] 能向非技术人员（老板/客户）清晰解释技术约束

## 二、实战场景决策树

### 场景 1：技术方案评审会上

**情境**：工程师提出技术方案，你需要判断合理性并提出问题。

#### 决策流程

```
工程师提出方案
    ↓
【步骤1】理解方案目标
    问：这个方案要解决什么问题？
    ↓
【步骤2】评估技术选型
    用下面的"5个关键问题清单"
    ↓
【步骤3】识别风险
    用"技术风险检查清单"
    ↓
【步骤4】确认可行性
    问：有 Baseline 吗？预期效果是多少？
```

#### 5个关键问题清单（根据方案类型选择）

**如果是 RAG 方案**：
1. **召回精度**："召回精度预计能到多少？有 Baseline 吗？"
2. **延迟影响**："Reranker 会增加多少延迟？用户能接受吗？"
3. **更新成本**："知识库更新频率怎样？全量重新 Embedding 成本是多少？"
4. **失败处理**："如果检索不到相关文档，怎么处理？"
5. **评估方法**："怎么评估 RAG 效果？有测试集吗？"

**如果是 Agent 方案**：
1. **任务边界**："Agent 能处理哪些类型的任务？哪些不能？"
2. **成本控制**："单次任务平均消耗多少 Token？有预算上限吗？"
3. **失败率**："预计失败率是多少？失败后怎么处理？"
4. **工具调用**："Agent 会调用哪些工具？工具失败怎么办？"
5. **可解释性**："用户能看到 Agent 的决策过程吗？"

**如果是模型选型**：
1. **能力匹配**："为什么选这个模型？对比过其他模型吗？"
2. **成本对比**："这个模型的成本是多少？有更便宜的替代方案吗？"
3. **延迟要求**："延迟能满足用户体验要求吗？（<3s）"
4. **上下文长度**："Context Length 够用吗？处理长文档会截断吗？"
5. **备选方案**："如果这个模型不可用（API 故障），有备选方案吗？"

#### 技术风险检查清单

在评审时，逐项检查以下风险：

- [ ] **幻觉风险**：高风险输出（医疗/财务/法律）有人工审核吗？
- [ ] **成本失控**：有 Token 预算上限和超时终止机制吗？
- [ ] **数据泄露**：用户数据会进入模型训练吗？（确认模型方的数据使用政策）
- [ ] **提示词注入**：有输入过滤和权限隔离吗？
- [ ] **性能瓶颈**：高并发下会不会崩溃？有压测结果吗？
- [ ] **依赖风险**：依赖的外部 API（模型/工具）挂了怎么办？

### 场景 2：向老板汇报技术风险

**情境**：发现技术风险，需要向老板汇报并提出应对方案。

#### 汇报话术模板

```
【风险描述】
我们在 [功能名称] 中发现 [风险类型]，具体表现为 [具体问题]。

【影响评估】
- 用户影响：[多少用户会受影响，影响程度]
- 业务影响：[对核心指标的影响，如转化率/留存率]
- 合规风险：[是否涉及法律/监管问题]

【应对方案】
我们计划采取以下措施：
1. [短期方案]：[具体措施]，预计 [时间] 完成
2. [长期方案]：[根本性解决方案]，预计 [时间] 完成

【资源需求】
需要 [人力/时间/预算] 支持。

【决策建议】
建议 [上线/延期/调整范围]，理由是 [权衡分析]。
```

#### 案例：幻觉风险汇报

❌ **错误示范**：
> "模型会出幻觉，我们没办法。"

✅ **正确示范**：
> "我们在医疗建议功能中发现模型有 5% 的概率会编造药品信息（幻觉问题）。这会影响约 1000 名/天的用户，存在医疗事故风险。
> 
> 我们的应对方案：
> 1. 短期（本周）：在所有医疗建议下方增加免责声明'AI 生成，请咨询医生'
> 2. 中期（2周）：增加人工审核环节，高风险输出（药品/剂量）必须经药师审核
> 3. 长期（1个月）：接入权威医疗知识库（RAG），降低幻觉率到 1% 以下
> 
> 建议先上线短期方案，中期方案需要 1 名药师兼职支持。"

### 场景 3：估算 AI 功能成本

**情境**：老板问"加个 AI 功能，成本怎样？"

#### 成本估算公式

```
日成本 = DAU × 使用率 × (输入Token × 输入单价 + 输出Token × 输出单价) / 1,000,000
月成本 = 日成本 × 30
```

#### 实战案例：AI 摘要功能

**需求**：为文章列表页增加 AI 摘要功能

**估算过程**：
```
假设：
- DAU 10万，30% 使用摘要功能 = 3万次/天
- 平均每次：3000 Token 输入（文章） + 500 Token 输出（摘要）
- 使用 GPT-4o-mini：$0.15/百万输入 + $0.6/百万输出

日成本 = 30,000 × (3000×0.15 + 500×0.6) / 1,000,000
       = 30,000 × (0.00045 + 0.0003)
       = 30,000 × 0.00075 
       = $22.5/天

月成本 = $22.5 × 30 = $675/月
```

**优化建议**：
- 如果成本太高，考虑：
  1. 只对 VIP 用户开放
  2. 限制每日使用次数（如 10 次/天）
  3. 使用更便宜的模型（如 Claude Haiku）
  4. 缓存热门文章的摘要（避免重复生成）

#### 成本估算速查表

| 模型 | 输入价格（$/百万Token） | 输出价格（$/百万Token） | 适用场景 |
|------|---------------------|---------------------|---------|
| GPT-4o | $2.5 | $10 | 复杂推理、代码生成 |
| GPT-4o-mini | $0.15 | $0.6 | 大多数场景 |
| Claude Sonnet | $3 | $15 | 长文本理解、分析 |
| Claude Haiku | $0.25 | $1.25 | 简单任务、高并发 |

## 三、技术知识优先级矩阵

### P0：必须掌握（面试必考，日常必用）

#### 1. 模型能力边界

**决策场景**：判断需求是否可行

| 模型能力 | 可用于 | 不适合 | 判断标准 |
|---------|--------|--------|---------|
| 文本生成 | 内容创作、代码补全、报告生成 | 需要精确数字计算 | 如果需要 100% 准确的数字，用传统算法 |
| 推理 | 分析、决策建议、诊断 | 要求 100% 准确率 | 如果错误不可接受（医疗/金融），必须人工审核 |
| 检索增强（RAG） | 知识库问答、客服 | 需要实时数据（分秒级） | 如果数据更新频率 < 1小时，考虑传统数据库 |
| 多模态 | 图片理解、UI 分析 | 视频实时流处理 | 视频处理成本极高，优先考虑关键帧提取 |

#### 2. 核心性能指标

**决策场景**：评估用户体验和成本

| 指标 | 含义 | 产品影响 | 合格标准 |
|------|------|---------|---------|
| Latency（延迟） | 首字符响应时间 | 用户体验，>3s 显著流失 | < 2s（优秀），< 3s（合格） |
| TTFT | Time to First Token | 流式输出的感知速度 | < 1s（优秀），< 2s（合格） |
| Throughput | 每秒生成 Token 数 | 高并发下的响应能力 | 根据 DAU 计算，确保不排队 |
| Context Length | 单次处理的最大 Token 数 | 决定能处理多长的文档 | 根据业务需求选择（32k/128k/200k） |
| Cost/Token | 每 Token 的费用 | 单次调用和月度成本 | 月成本 < 预算的 70%（留缓冲） |

#### 3. 常见技术风险及应对

| 风险类型 | 具体表现 | PM 应对策略 | 验收标准 |
|---------|---------|-----------|---------|
| 幻觉（Hallucination） | 模型编造事实 | 增加来源引用、人工审核高风险输出 | 高风险场景幻觉率 < 1% |
| 提示词注入 | 用户通过输入劫持模型行为 | 输入过滤、权限隔离 | 通过红队测试 |
| 成本失控 | 复杂 Agent 任务 Token 暴涨 | 设置 Token 预算上限、任务超时终止 | 单次任务成本 < $0.1 |
| 数据泄露 | 用户数据进入模型训练 | 确认模型方不用 API 数据训练 | 签署数据使用协议 |

### P1：建议掌握（提升决策质量）

#### 4. 部署模式选择

**决策场景**：技术选型时的权衡

| 模式 | 特点 | 适用场景 | 成本对比 |
|------|------|---------|---------|
| API 调用（云端） | 低成本、快速上线 | 大多数产品 | 按量付费，灵活 |
| 私有化部署 | 数据不出域，成本高 | 金融/医疗等合规要求高 | 固定成本高（服务器+运维） |
| 边缘推理（本地） | 低延迟、隐私好，性能有限 | 手机 App、离线场景 | 无 API 成本，但模型能力受限 |
| 混合部署 | 本地小模型 + 云端大模型 | 成本与性能平衡 | 复杂度高，需要路由策略 |

**决策树**：
```
是否有严格的数据合规要求（金融/医疗）？
├─ 是 → 私有化部署
└─ 否 → 是否需要离线使用？
    ├─ 是 → 边缘推理（本地模型）
    └─ 否 → 是否有极高的并发量（>10万QPS）？
        ├─ 是 → 混合部署（降低成本）
        └─ 否 → API 调用（最简单）
```

## 五、技术沟通能力进阶

### 与工程师沟通的黄金法则

1. **提问前先做功课**：不要问"RAG 是什么"，而是问"我们的 RAG 召回率为什么只有 60%？"
2. **用数据说话**：不要说"延迟太高"，而是说"P95 延迟 5s，超过用户可接受的 3s"
3. **承认不懂**：不要装懂，直接说"这个技术细节我不太懂，能解释一下吗？"
4. **记录并结构化**：每次沟通后，把结论整理成文档，方便后续查阅

### 向非技术人员解释技术的技巧

1. **用类比**：把 Token 比作"字数"，把 Context Length 比作"短期记忆容量"
2. **避免术语**：不要说"Embedding 向量化"，而是说"把文本转换成计算机能理解的数字"
3. **强调影响**：不要说"模型会幻觉"，而是说"模型有 5% 概率会编造事实，影响用户信任"
4. **给出方案**：不要只说问题，要同时给出解决方案和资源需求

## 六、面试高频问题及回答策略

### Q1：你的技术背景怎样？怎么弥补技术短板？

**回答策略**：展示"够用原则" + 持续学习

> "我的策略是'够用原则'：深入理解能力边界和风险，而不是实现细节。我会定期动手做原型（用低代码平台或 API），确保对技术有感性认知。
> 
> 举个例子，上个月我用 Dify 搭了一个 RAG 知识库，亲自体验了召回精度和延迟的权衡，这让我在评审技术方案时更有底气。
> 
> 遇到不懂的，我会直接问工程师，并把结论结构化后带回产品决策。比如我会问'为什么选 Reranker 而不是直接用 Embedding 排序？'，然后把答案整理成决策文档。"

### Q2：你会写代码吗？

**回答策略**：承认能力边界 + 强调 PM 价值

> "我能写基础的 Python 和 JavaScript，可以用 Claude/GPT API 搭简单原型来验证想法。但我不会把精力花在成为工程师上——PM 的价值在于定义'做什么'和'为什么做'，而不是'怎么做'。
> 
> 我的代码能力主要用于：
> 1. 快速验证产品想法（搭原型）
> 2. 理解工程师的技术方案（能看懂代码逻辑）
> 3. 估算技术成本和风险（能算 Token 消耗）"

### Q3：如何评估一个技术方案的合理性？

**回答策略**：展示结构化思维 + 实战案例

> "我会从四个维度评估：
> 
> 1. **目标匹配度**：这个方案能解决我们的核心问题吗？有没有过度设计？
> 2. **技术风险**：有哪些潜在风险？应对方案是什么？
> 3. **成本可控性**：成本在预算内吗？有没有优化空间？
> 4. **可维护性**：上线后好维护吗？依赖的外部服务稳定吗？
> 
> 举个例子，上次工程师提出用 Multi-Agent 实现客服系统，我发现：
> - 目标匹配度：80% 的问题是简单查询，不需要 Multi-Agent
> - 技术风险：Agent 失败率 10%，用户体验差
> - 成本：单次对话成本 $0.5，月成本超预算 3 倍
> 
> 最后我们改用 RAG + 简单规则引擎，成本降低 80%，效果反而更好。"

## 附录：常用技术术语速查表

| 术语 | 含义 | PM 需要知道的 |
|------|------|-------------|
| Token | 文本的最小单位（约 0.75 个英文单词） | 成本和延迟的计算单位 |
| Context Length | 单次处理的最大 Token 数 | 决定能处理多长的文档 |
| Temperature | 控制输出随机性的参数（0-2） | 0=确定性，1=平衡，2=创造性 |
| Embedding | 把文本转换成向量 | RAG 的核心技术 |
| Reranker | 对检索结果重新排序 | 提升 RAG 精度，但增加延迟 |
| Fine-tuning | 在特定数据上微调模型 | 成本高，适合垂直领域 |
| Prompt Engineering | 优化提示词以提升效果 | 最低成本的优化方式 |
| Hallucination | 模型编造事实 | 高风险场景必须应对 |
| Latency | 响应时间 | 用户体验的关键指标 |
| Throughput | 每秒处理请求数 | 高并发场景的瓶颈 |
