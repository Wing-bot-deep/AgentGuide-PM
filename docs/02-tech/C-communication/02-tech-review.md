# 技术方案评审

> 评审会上沉默的 PM 和乱提意见的 PM 一样危险——前者被忽视，后者被排斥。

---

## 评审会前：准备工作

### 提前读文档（别等会上才看）

收到技术方案文档后，提前用"5分钟扫描法"（见[读懂代码](../A-coding/01-read-code#三读技术方案文档)）过一遍：

- 圈出看不懂的词，会前问清楚（别在会上暴露）
- 标记与 PRD 有出入的地方，准备对齐
- 想好你关心的 2-3 个核心问题

### 准备你的视角

工程师关注**可行性**，PM 关注**用户体验 + 业务目标**。

会上你的核心任务是：

1. 确认方案能达成产品目标
2. 识别用户感知到的风险
3. 确认上线策略和评估方案

---

## 评审会上：提问框架

### 五类有效问题

**类型 1：目标对齐**
> "这个方案能解决 [用户痛点 X] 吗？最弱的环节在哪里？"

**类型 2：用户感知**
> "从用户视角，最坏情况下的体验是什么？出现概率多大？"

**类型 3：边界条件**
> "[边界场景 X] 时，用户会看到什么？"

**类型 4：可观测性**
> "上线后，我们怎么知道这个方案是否在正常工作？有哪些监控指标？"

**类型 5：回退能力**
> "如果上线后发现有问题，回滚的代价是什么？需要多久？"

## AI 产品技术方案的特殊检查点

### Prompt 相关

```
□ Prompt 是否已经过测试？测试集有多大？
□ Prompt 版本如何管理？改 Prompt 算需要发版吗？
□ 不同模型版本的 Prompt 是否需要分开维护？
□ Prompt 中有没有写死的信息（如日期、版本号）会随时间失效？
```

### 模型选型

```
□ 为什么选这个模型？和其他选项的对比依据是什么？
□ 模型供应商的 SLA 是多少？历史可用性如何？
□ 如果供应商涨价或停服，切换成本是多大？
□ 是否考虑过本地部署作为备选？
```

### 性能与成本

```
□ 预计 P50 / P99 延迟是多少？
□ 高并发下的降级策略是什么？（排队 / 限流 / 切换小模型）
□ 单用户月均成本估算是多少？盈利模型是否 cover？
□ Token 消耗监控怎么做？有没有异常告警？
```

### 安全与合规

```
□ 用户输入是否会发送给第三方模型供应商？用户知道吗？
□ 模型供应商是否会用 API 数据训练模型？（合规敏感）
□ 有没有输入过滤？防止提示词注入？
□ 输出有没有安全过滤？（有害内容、隐私信息泄露）
```

### 评估与迭代

```
□ 上线的成功标准是什么？如何测量？
□ 有没有 AB 测试方案？样本量够吗？
□ 用户反馈如何收集？（隐式：点赞点踩 / 显式：评分）
□ 模型效果变差时，怎么发现？（是否有质量监控）
```

## 常见对话场景

### 场景 1：工程师说"这个 AI 效果不好保证"

不好的回应：
> "那你们想办法让它好"（推卸责任）

好的回应：
> "我们先定义什么叫'够好'——我来提供测试集和评分标准，你们来跑 Baseline，一起决定是否达到上线门槛。"

### 场景 2：工程师说"这个功能要 3 周"

不好的回应：
> "能不能快点？" / "其他公司 1 周就做出来了"

好的回应：
> "帮我理解一下 3 周的主要工作量在哪里？有没有可以拆成 MVP 版本先上的部分？"

### 场景 3：工程师说"你的需求描述不清楚"

不好的回应：
> "我写了那么多页了还不清楚？"

好的回应：
> "帮我具体说哪里不清楚，我现在就补充。"（然后立刻更新 PRD，并在会上确认）

### 场景 4：评审会上出现分歧

不要在会上强行决策。做法：

1. 明确分歧点："我理解你们的顾虑是 X，我的诉求是 Y，对吗？"
2. 拆解出决策依据："我们需要哪些数据才能决定？"
3. 明确后续行动："谁在什么时间提供这个数据？下次会议再定。"

## 与工程师建立长期信任

技术沟通是长期关系，不是单次博弈：

**建立信任的行为**：
- 承诺的 PRD 准时交付，且质量高
- 会上提的问题基于真实用户反馈，而非个人感受
- 工程师提出技术约束时，认真理解而不是立刻反驳
- 功能上线后，主动分享数据结果（不只在出问题时找他们）

**破坏信任的行为**：
- 需求频繁变更，且没有解释原因
- 在评审会上当众质疑工程师判断
- 越过负责工程师直接找 leader 施压
- 只在出问题时沟通，平时不回消息
