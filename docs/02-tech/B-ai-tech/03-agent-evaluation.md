# Agent 评估体系

> 原文：《AI Agent 评估完全指南》，来源：[AgentGuide](https://github.com/adongwanai/AgentGuide) / Anthropic 技术文章。
> 本文为 AI PM 必掌握的核心技能：如何知道你的 Agent 产品到底好不好。

---

## 核心问题：为什么传统测试方法失效？

传统软件测试：输入确定 → 输出确定 → 断言通过 ✅

AI Agent 有两个让传统测试失效的特性：

### 特性 1：非确定性

同样的输入，今天对明天错。模型版本、上下文、采样参数的微小变化都会影响输出。你不能用 `assert output == expected` 来测 Agent。

### 特性 2：多轮级联错误

```
第1轮: 读取订单  ✅
第2轮: 计算退款（算错了 1 分钱）✅
第3轮: 更新数据库  ✅
...
第10轮: 生成财务报表  ❌ 对不上账
```

第 2 轮的小错误，在第 10 轮引发大问题。早期错误会沿着多轮交互不断放大。

## 评估系统的四个核心组件

### 1. Task（任务）：不是 Prompt，是"考题"

真正的 Task 需要有：
- 明确的初始状态（给什么数据）
- 明确的成功标准（完成是什么样的）
- 预期结果（可以用来打分）

```
❌ 简单 Prompt："帮我订机票"
✅ 标准 Task：
   - 初始状态：用户信息 + 航班数据库
   - 成功标准：数据库有订单记录 + 用户收到确认邮件
   - 预期结果：指定航班 + 座位号 + 价格
```

### 2. Trial（试验）：跑一次不算，要跑多次

因为非确定性，单次测试可能是运气好/运气差：

| 用途 | 建议次数 |
|------|---------|
| 快速验证 | 3 次 |
| 正式评估 | 10 次 |
| 关键功能上线前 | 100 次 |

有了多次试验，才能得到统计意义上的成功率。

### 3. Grader（评分器）：需要多个维度打分

| 类型 | 方式 | 适用场景 |
|------|------|---------|
| **代码评分** | 精确判断（断言、正则）| 格式验证、数值计算 |
| **模型评分** | 用更强的 LLM 判断 | 语义质量、逻辑合理性 |
| **人工评分** | 专家标注 | 最终基准，用于校准模型评分 |

三种评分器组合使用，形成"瑞士奶酪"式防御——每层有漏洞，但组合起来几乎无懈可击。

### 4. Transcript vs. Outcome：说的 vs. 做的

| | Transcript（日志） | Outcome（结果） |
|---|---|---|
| **是什么** | Agent 的执行过程 | 环境的真实变化 |
| **可信吗** | ❌ Agent 可能报告错误状态 | ✅ 数据库不会说谎 |
| **用来干嘛** | Debug、分析行为 | 验证任务是否完成 |

**重要**：一定要同时看 Outcome（实际结果），而不只看 Transcript（Agent 说它做了什么）。Agent 可能报告"成功"但实际上什么都没发生。

## 处理非确定性：pass@k vs. pass^k

面对随机性，需要选择合适的评估模式：

| 指标 | 含义 | 适用场景 |
|------|------|---------|
| **pass@k** | k 次中至少 1 次成功 | 有人工验证，偶尔成功可接受 |
| **pass^k** | k 次全部成功 | 全自动流程，必须稳定 |

**PM 决策**：你的产品是"有人监督的工作流"还是"全自动执行"？这决定了用哪个指标。

## 四类 Agent 的评估重点

| Agent 类型 | 核心评估重点 |
|-----------|-----------|
| **代码 Agent** | 代码能否正确运行；有无安全漏洞；测试通过率 |
| **文档处理 Agent** | 信息提取准确率；关键内容是否遗漏；格式是否正确 |
| **工作流 Agent** | 任务完成率；步骤正确性；错误恢复能力 |
| **对话 Agent** | 回答相关性；一致性（前后不矛盾）；拒绝不当请求 |

## AI PM 的三层评估体系

结合产品视角，Agent 评估应该覆盖三层：

### 第一层：模型层（离线评估）

- **任务完成率**：核心功能的成功率
- **准确率**：结果是否正确
- **延迟**：响应时间（TTFT + 完整生成时间）
- **Bad Case 集合**：持续回归测试

### 第二层：产品层（在线监控）

- **用户接受率**：用户是否采纳了 Agent 的建议/操作
- **追问率**：用户是否因为结果不满意而重新提问
- **手动纠错率**：用户多频繁修改 Agent 的输出
- **任务完成率**：用户启动的任务最终有多少真正完成

### 第三层：业务层（结果验证）

- **效率提升**：使用 Agent 后的任务耗时变化
- **错误率变化**：和人工操作相比，Agent 的错误率
- **用户满意度**：NPS / CSAT

## 从零开始的评估路线图

```
第一步：20 个测试用例（先有再好）
         ↓
第二步：明确每个用例的成功标准
         ↓
第三步：用代码评分实现自动化
         ↓
第四步：加入模型评分覆盖模糊场景
         ↓
第五步：建立 Bad Case 收集闭环
         ↓
第六步：每次版本迭代前跑完整测试套件
         ↓
持续：新 Bad Case → 加入测试集 → 修复 → 回归
```

**关键原则**：宁愿有 20 个粗糙的测试用例，也不要等待"完美的评估框架"。先动起来，逐步迭代。

## PM 常见问题

**Q：评估是工程师的事，PM 需要懂这些吗？**

A：AI PM 需要能够提出评估方案、解读评估结果、基于数据做优先级决策。不需要自己写代码，但必须能和工程师对齐"什么是成功"。

**Q：指标看起来很好，但用户投诉怎么办？**

A：说明产品层指标（用户行为）和模型层指标（自动评估）不一致。需要回到产品层，重新设计更贴近用户真实体验的评估指标。

**Q：预算有限，评估从哪里开始？**

A：从最核心的 1-2 个功能开始，各建 10 个测试用例，用代码评分实现自动化。成本极低，但能快速发现大多数问题。
