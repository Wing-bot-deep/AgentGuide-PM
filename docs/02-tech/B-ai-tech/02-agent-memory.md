# Agent 记忆系统设计

> 原文：《Agent Memory - 从原理到实战》，来源：[AgentGuide](https://github.com/adongwanai/AgentGuide)。
> 本文为 PM 视角改写，重点关注产品设计而非代码实现。

---

## 为什么 Agent 需要记忆？

没有记忆的 Agent，每次对话都是全新开始——用户今天告诉你"我喜欢简洁风格"，明天又要重新说一遍。这不是助手，是金鱼。

记忆系统让 Agent 能够：
- **持久化用户偏好**：跨会话记住用户习惯
- **任务连续性**：长期任务在多轮对话中保持状态
- **知识积累**：随着使用时间增长，越来越懂用户
- **个性化服务**：基于历史行为定制响应

## 三层记忆架构

```
┌─────────────────────────────────────┐
│   第一层：工作记忆（Working Memory）    │
│   当前会话的对话历史                    │
│   存储：内存（最快）                    │
│   容量：有限（最近 10-20 轮）           │
└─────────────────────────────────────┘
            ↓ 重要信息提取
┌─────────────────────────────────────┐
│   第二层：情节记忆（Episodic Memory）   │
│   本次会话的关键信息                    │
│   存储：向量数据库                     │
│   容量：中等（单会话级别）               │
└─────────────────────────────────────┘
            ↓ 知识提炼
┌─────────────────────────────────────┐
│   第三层：语义记忆（Semantic Memory）   │
│   用户长期偏好、历史知识                │
│   存储：知识图谱 + 向量库              │
│   容量：无限（跨会话持久化）            │
└─────────────────────────────────────┘
```

**PM 视角类比**：
- 工作记忆 = 当前打开的浏览器标签页（快速但有限）
- 情节记忆 = 最近的浏览历史（可回溯）
- 语义记忆 = 书签收藏夹（永久保存的重要内容）

## 五个核心设计问题

在产品设计阶段，需要回答这五个问题：

### 1. 什么时候存储？

不是所有对话内容都值得记忆，触发存储的条件：
- 用户明确提到偏好（"我喜欢…"、"以后记住…"）
- 用户确认了重要决策
- 任务完成时的关键状态
- 用户纠错时（负反馈也是记忆）

### 2. 用什么格式存储？

| 格式 | 适用场景 | 示例 |
|------|---------|------|
| 原始文本 | 对话内容保真回放 | 完整对话记录 |
| 结构化摘要 | 快速检索用户属性 | `{name: "张三", 偏好: ["简洁", "中文"]}` |
| 向量 Embedding | 语义相似度检索 | 历史对话的向量表示 |
| 知识图谱 | 实体关系密集的场景 | 用户-偏好-场景三元组 |

### 3. 如何检索？

- **关键词检索**：快，适合精确匹配
- **向量相似度检索**：语义匹配，适合"找相关记忆"
- **混合检索**：两者结合，兼顾准确性和语义理解

### 4. 如何遗忘？

记忆系统不能无限增长，常见的遗忘策略：
- **时间衰减**：越旧的记忆权重越低
- **频率优先**：经常被访问的记忆优先保留
- **显式删除**：用户主动请求删除（GDPR 合规必须支持）
- **容量限制**：超过阈值时删除最旧/最少访问的记忆

### 5. 如何处理矛盾信息？

用户今天说"喜欢中文"，明天说"喜欢英文"，两条记忆冲突怎么办？
- **时间优先**：新信息覆盖旧信息（最常用）
- **置信度加权**：多次确认的信息权重更高
- **用户确认**：当矛盾时主动询问用户

## 三种主流实现方案

### 方案 1：Mem0（最简单，推荐入门）

**适合场景**：对话 Agent、个性化推荐、快速原型

**核心优势**：
- 自动实体提取（不需要手动写提取逻辑）
- 自动去重（同一信息不会存两次）
- 支持多种向量数据库后端

**适合 PM 了解的点**：
- 10 行代码就能集成，工程门槛很低
- 信息提取依赖 LLM 调用，有额外 Token 成本
- 生产环境需要更多测试

---

### 方案 2：MemGPT（虚拟内存机制）

**适合场景**：长期对话、复杂任务、上下文窗口受限的场景

**核心思路**：模仿操作系统的虚拟内存——工作中的信息放"内存"（上下文），不用的信息放"磁盘"（外部存储），需要时换入。

**适合 PM 了解的点**：
- 解决"上下文窗口有限但对话很长"的根本矛盾
- 对用户透明——用户感觉 Agent 一直"记得"，但实际上在动态换页
- 延迟比 Mem0 高（需要额外的内存管理操作）

---

### 方案 3：自定义实现（LangChain）

**适合场景**：有特殊需求、需要精细控制记忆策略的团队

**适合 PM 了解的点**：
- 最灵活，但工程成本最高
- 适合把记忆系统作为核心竞争力的产品（如个人 AI 助手）
- 需要工程团队深度参与设计

## 记忆系统的产品设计要点

### 透明度设计

用户应该能够：
- **查看**：我的 AI 记住了哪些内容
- **编辑**：修改不准确的记忆
- **删除**：清除特定记忆或全部记忆
- **关闭**：选择不使用记忆功能

这既是产品体验的要求，也是 GDPR / 《个人信息保护法》的合规要求。

### 隐私边界

- 敏感信息（医疗、财务、密码）不要放入长期记忆
- 多用户场景下，记忆隔离是基本要求，不能串用
- 记忆数据需要和普通对话数据一样受到保护

### 成本控制

记忆系统会带来额外的 Token 消耗：
- 每次对话都要检索相关记忆（检索 Token）
- 自动提取和存储时需要 LLM 调用（写入 Token）
- 压缩摘要时需要 LLM 调用（压缩 Token）

**PM 建议**：在产品设计时，把记忆功能的 Token 成本单独列出来，纳入产品成本模型。
